# report: fetch_stock.py 株価系列の分割調整(AdjC/AdjVo基準)恒久パッチ(2026-07-07)

## 背景
1980ダイダン(過去)・3110日東紡績(2026-06-29の1:5分割)で、fetch_stock.pyの52週高値/安値・EMA13/25/75が分割未調整の生値(C)から算出され、桁違いの値(例: 3110で「52週高値31,950円」vs実勢3,515円)を返す不具合が2回発生。いずれもノート内の手動補正(AdjC再計算)で凌いでいたが、注意力依存のため恒久修正した。

## 原因
`fetch_stock.py`の列選択ロジックが `next(c for c in ["C", "Close", "AdjC", "AdjustmentClose"] if c in p.columns)` という優先順位で、生値`C`を`AdjC`より先に選んでいた。J-Quantsの `get_eq_bars_daily` は `C`(生値)と`AdjC`(分割調整後値)を両方返すため、`C`が存在すれば常にそちらが採用され、株式分割をまたぐ期間で不連続な系列になっていた。

## 修正内容(fetch_stock.pyのみ変更)
- 列選択の優先順位を `["AdjC", "AdjustmentClose", "C", "Close"]` / `["AdjVo", "AdjustmentVolume", "Vo", "Volume"]` に変更(調整後列を優先)
- 現在値の調整後列/生値が不一致の場合の警告ログを追加(通常は一致するはずで、不一致は想定外の挙動として記録・採用は常に調整後列)
- 出力に `price.note: "株価系列は分割調整後(AdjC基準)"` を追加
- **算出方法自体は変更していない**: 52週レンジは従来通り終値ベース(高値/安値ベースではない)、EMAは従来通り`ewm(span=X)`の既定パラメータ(adjust=True)のまま。変更したのは「どの列を終値として使うか」のみ

## 検証結果

### 3110日東紡績(分割銘柄)
修正後: 52週安値1,060円(2025-08-20)・高値6,390円(2026-05-07)・EMA13=3,911.6/EMA25=4,069.4/EMA75=4,273.2。いずれも現在の株価水準(3,515円)と整合する桁になった(修正前は31,950円等の桁違いの値だった)。

**既存ノート(3110_日東紡績_2026-07-07)記載の手動再計算値(52週安値1,050円・高値6,580円)とは数%の差異あり。原因を特定**: ノート作成時の手動計算では52週高値・安値を「高値(AdjH)・安値(AdjL)」ベースで算出していたが、fetch_stock.pyの52週レンジは元々(バグ修正前から)「終値(C)」ベースの設計だったため、修正後も終値ベースを維持した(1980の既存手動補正がこの終値ベース設計と一致していたため、設計の一貫性を優先)。結果として3110ノートの手動値(高値/安値ベース)と今回の修正後出力(終値ベース)に3110固有で約2-3%の差異が生じる。これは「丸め差」ではなく算出方法の違いによるものだが、原因が明確で軽微(3110の場合、高値6,580円は寄り付き高値、終値ベースの高値は6,390円)なため、算出方法(終値 vs 高値/安値)を統一する場合は別途明示的な追加変更が必要である旨を報告する。EMA75の微差(4,273.2 vs ノート記載4,270.7)は`ewm`のadjust=True/False規約の違いによるもので実務上無視できる差。

### 1980ダイダン(分割銘柄、過去ノートとの整合確認)
修正後の52週高値3,670円(2026-02-27)・安値1,443.3円(2025-07-11)は、既存ノート(1980_ダイダン_2026-07-05)がAdjCベースで手動再計算した値と完全一致(1980の手動補正は元々終値ベースだったため、今回の修正方針と自然に一致した)。EMAはわずかな差(数営業日分のlookback差・adjust規約差)のみ。

### 6637・9616(分割なし銘柄、回帰確認)
`git stash`による変更前後比較で、`price.note`追加以外は完全一致(w52_high/low・EMA・avg_vol_20d・latest_close等すべて同一)。回帰なしを確認。

## 付帯調査(修正はせず記録のみ、指示通り)
同型の「生値`C`を`AdjC`より優先する」列選択パターンを`dashboard2/build_data2.py`(line 148、port8082「セカンドダッシュボード」の`fetch_jp`関数)に発見した。ただしこちらは現値/前日比/出来高の算出にのみ使われ、52週高値/安値・EMAのような複数週にまたがる集計は行っていないため、影響は「分割が発生した当日の前日比計算が1日だけ誤る」程度に限定される(fetch_stock.pyの不具合のような桁違いの持続的な誤りにはならない)。今回のタスク範囲(fetch_stock.pyのみ変更)outのため修正はしていない。なお`alert_system.py`は既に`AdjC`を優先する実装になっており、同型の不具合は無いことを確認した。
